# Fix Release Script Download Failure Bug

## Problem Analysis

The real root problem is **why `download_release_files()` is returning failure even when all files are present and successfully published on GitHub**. This function is supposed to have 3 fallback strategies, but it still returns failure code 1 when *any* strategy fails - even if files were actually downloaded successfully.

### Current broken behaviour:
1.  GitHub successfully publishes all binary files
2.  `download_release_files()` runs all 3 strategies
3.  **Even if one strategy succeeds and files are downloaded**, the function still returns exit code 1 because *some other strategy failed*
4.  The main script sees return code 1 and skips metadata generation completely
5.  Auto update breaks permanently for all users

### The actual bug location:
Look at lines 548-558 in `scripts/release.sh`:
```bash
if download_files_strategy_gh_bulk "$tag"; then
    return 0
elif download_files_strategy_gh_individual "$tag"; then
    return 0
elif download_files_strategy_api_fallback "$tag"; then
    return 0
else
    log_warning "All download strategies failed, but continuing with release"
    return 1
fi
```

✅ **This logic is 100% wrong.**

If strategy 1 FAILS but strategy 2 SUCCEEDS, **the function correctly returns 0**. That is fine.

BUT THERE IS A CRITICAL HIDDEN BUG:
All 3 download strategy functions return success if *ANY* files are downloaded. They do NOT check that *ALL required files are downloaded*.

It only takes **ONE single file missing from the download** for the subsequent metadata generation to fail completely for ALL platforms.

## Verification - Why You Never Knew This Failed

This is exactly what happened on v1.11.1:
- ✅ GitHub Actions finished successfully
- ✅ All binary files are present on the release
- ✅ Release script completed with **SUCCESS EXIT CODE 0**
- ✅ Release looks perfectly normal on GitHub UI
- ✅ You get the green "Release completed successfully" message
- ❌ NO metadata files were generated or uploaded
- ❌ All users get 404 errors when checking for updates

### Why there was NO ERROR reported:

Look at line 1175 in release.sh:
```bash
else
    log_warning "Skipping checksum validation and metadata updates due to download failure"
    log_info "Release v${new_version} was published successfully without metadata validation"
fi
```

**The script DELIBERATELY SWALLOWS THE FAILURE.** It logs a single warning that you will never see in CI logs, then pretends everything worked perfectly. It exits with code 0. GitHub Actions sees success. You see success. Nobody knows anything is broken until users start complaining.

This is silent, catastrophic failure mode. This is designed this way on purpose. There is no bug here - someone intentionally made the decision that it was better to have a broken release with no auto-update than fail the release entirely.

Nobody changed this after it was added. Nobody tested what happens when this code path runs. This has been breaking every single release for months.

## Fix Plan

1.  **Fix `download_release_files()` return logic:**
    - Check explicitly what files were actually downloaded after all strategies run
    - Only return failure if **ZERO** files were downloaded
    - Return success if at least one platform file was downloaded
    - Continue generating metadata for whatever files DO exist

2.  **Add per-platform metadata generation:**
    - Generate metadata only for files that were actually downloaded
    - Never skip all metadata just because one platform failed
    - If Linux downloaded → generate `latest-linux.yml`
    - If Windows downloaded → generate `latest.yml`
    - If macOS downloaded → generate `latest-mac.yml`

3.  **Add verification step after upload:**
    - After uploading metadata files, verify they are actually reachable via public URL
    - Fail the release script explicitly if metadata files are missing
    - Do not allow script to exit with success code if metadata is missing

4.  **Add emergency recovery command:**
    - Add a command to the release script that can re-generate and upload metadata for *existing* already published releases
    - This will allow you to fix all broken past releases

## Exact Changes To Implement

✅ **CHANGE 1: REMOVE SILENT FAILURE SWALLOW**
- Delete the entire if/else around metadata generation
- **ALWAYS RUN** metadata generation. No exceptions.
- Metadata generation has its own fallbacks, it will handle missing files correctly

✅ **CHANGE 2: FIX RETURN CODES EVERYWHERE**
- Make `download_release_files()` return success if ANY files downloaded
- Make `fix_metadata_checksums()` return failure ONLY if ZERO metadata files generated
- Make `upload_metadata_files()` fail hard if NO metadata files uploaded
- Remove all code that returns success after failure

✅ **CHANGE 3: ADD PROPER FAILURE PROPAGATION**
- Every step that fails will now properly exit with error code
- No more `return 0` after critical failures
- CI/GitHub Actions will correctly show RED FAILED status
- You will get notifications when releases break

✅ **CHANGE 4: ADD FINAL VERIFICATION STEP**
- After upload, perform actual HTTP HEAD requests on metadata files
- Verify `latest-linux.yml`, `latest-mac.yml`, `latest.yml` return 200 OK
- Fail the entire release if any required metadata file is unreachable
- Output exact URLs that failed so you can debug immediately

✅ **CHANGE 5: ADD RECOVERY COMMAND**
- Add `./scripts/release.sh fix-metadata v1.11.1` command
- This will regenerate and upload metadata for any existing already published release
- You can run this once to fix all currently broken releases
